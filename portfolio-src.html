Deal Desk Engine

01
live

I built a deal desk tool that takes a deal, scores it on margin and discount discipline, and either auto-approves it, routes it to the right approver, or rejects it. The decisions are deterministic, so you can trust and audit them. AI structures messy deal notes, writes the plain-English explanation, and drafts the term sheet. It never makes the decision.

The sharpest beat: a deal at 78% margin still routes, because one line was discounted 35% past a per-line floor. Whether a deal makes money and whether a rep is holding the line on price are two different questions. The tool checks both, separately.

Deals route by reason, not one yes-or-no: Net-90 terms go to Finance, a termination-for-convenience clause goes to Legal. Every decision shows its work (inputs, math, rule fired) and every threshold is config, not code, so RevOps edits a number when policy changes and the logic never moves. The decision engine calls no model, so the expensive 80% is free; three AI edges cost pennies per deal.

Built as an interview case study. Runs entirely on synthetic pricing data.

deal-brain

02
in build · architecture locked

RevOps teams lose deals twice: once when the deal dies, again when the reason dies with it. Closed-lost dropdowns say "price" while the call transcripts say "no champion." The answer to "what's really happening on this deal?" is spread across CRM fields, field history, transcripts, emails, and Slack. 8 calls, 1 question, 0 shared context.

deal-brain does the plumbing once: everything syncs into one pre-joined Postgres store with a SFDC-shaped schema (including field history, the thing forecast analytics actually runs on), one get_deal_context(deal) interface, and three agents on top. A 2 to 3 hour weekly ritual becomes a 5-minute Slack brief.

All data is generated, seeded, and committed, with a held-out ground-truth manifest, so accuracy gets measured and published, including the misses. Zero real deal data, ever.

Claude OS + Hermes

A personal second brain plus an always-on agent stack: a desktop, a laptop, and a VPS, syncing through GitHub as the single hub, reachable from my phone over Telegram. It is not a demo. It runs my mornings, my research, my food, my training, and a weekly newsletter, every day, unattended.

Cross-platform architecture

01

Content never travels machine-to-server directly. Everything meets at GitHub; live queries take a separate path over a private VPN (Tailscale). Data is tiered by sensitivity, and the tiers are walls, not conventions.

Multi-model by design. Claude builds. Every high-consequence change is critiqued by a second model family (Codex) before it ships, because blind spots are correlated: the verifier must be independent of the source. Gemini is the third engine, used for 1M-context whole-vault reads and as a tiebreaker when the first two disagree. The VPS agents run their own model with an off-provider fallback. No single vendor is a point of failure, or an unchecked authority.

The ingest engine

03

The capture rule: zero-structure input, automated organization. Nothing asks me to file, tag, or format at capture time. I send a link, save a post, or clip a page; the pipeline does the rest.

Untrusted web content is treated as hostile until proven otherwise. The autonomous researcher is tool-less: it can only write inert markdown. A dual-LLM firewall sanitizes and classifies every capture, a deterministic regex scan quarantines anything carrying injection markers before a model ever reads it, and a human gate sits before the wiki. Prompt injection is assumed, not hoped away.

The wiki itself is Karpathy-style: flat markdown, no RAG, no embeddings, pages organized by function (sources, patterns, concepts, guides) with per-domain verdict pages that record what to adopt and what to skip.

Memory + optimizations

04

The unglamorous layer that keeps the system cheap, legible, and durable. Most of it is subtraction: no vector database, no embeddings, no framework, flat files an agent (or I) can read directly.

llm wikiThe second brain is a Karpathy-style wiki: flat markdown, no RAG, no embeddings, links resolve by filename, so it survives model and tool changes. An earlier semantic search index was built, measured, and retired once flat-file navigation proved sufficient.
okf indexesEvery managed folder carries an index.md describing its contents, so an agent loads cheap folder metadata before opening files. Navigation cost drops before a single document is read.
deterministic recallWiki lookups run through a scoring retriever (index lines + filename tokens, no model in the loop) that opens only the top page. About 60% cheaper and 55% faster than letting the model read around.
headroomAn always-on compression proxy sits between the editor and the API, compressing conversation context in flight. Long build sessions stay affordable without changing how I work.
model routingTwo operating points, nothing in between: an everyday model for routine work, escalation to the frontier the moment a trigger fires (irreversibility, trust boundaries, ambiguous requirements). Consequence beats category.
review dispatchSecond-model critiques run in an isolated subagent, so the bulky review transcript is billed once instead of riding along in every later turn (about 18% measured saving).
continuityEvery session ends in a written handoff and starts warm from it; an append-only session log is the continuity record. A cold start reads one file, not a history.

Security posture

05

The operating principle: capabilities are built but fenced. Every action sits behind an approval gate, a live flag, or a scoped token until I deliberately open it.

exposureThe VPS runs non-root; services publish on the private VPN address only. The sole public listener is key-only SSH. A Docker egress allowlist blocks abuse ports.
secretsKeys live only in gitignored env files, isolated per component in a container the agents cannot read, with a written rotation runbook.
trust gatesA second, independent model family critiques every hard-to-reverse change before it ships. Calendar writes take a two-phase confirm. Hooks block destructive git operations.
piiThe most sensitive tier never leaves the laptops. People's names are masked in the database. The EA is excluded from any fallback provider.
untrusted inputWeb content is fenced by the ingest pipeline above: tool-less researcher, dual-LLM firewall, deterministic quarantine, human gate.

Build your own

06

I distilled the whole build into a starter kit: the build order (seven steps, each buying a specific capability), a phased VPS setup runbook with a security checklist, de-personalized templates for the context files, a secrets-101 guide, and copy-paste prompts that point your own Claude Code at each step.

It exists because three of the seven steps are where everyone gets stuck, and the kit puts a verification at the end of every risky step: prove the gitignore holds before the first push, prove the VPN wall refuses outside connections before going always-on.
